## Security Validation Report — `engine/event-emitter.ts`

### Findings Validation

#### SEC-1 — URL path injection via `sessionId` and `agentId` → **AGREE: P1, but downgrade to P3 in practice**

**Confirmed** that `sessionId`, `opts.sessionId`, and `pgId` are interpolated raw into URL paths at lines 255, 274, 286, and 304.

**However, the actual risk is effectively nil in the current codebase:**

1. **`sessionId`** is always generated by `randomUUID()` at `orchestrator.ts:199`. It's never user-supplied. There's no code path where an external caller controls the session ID flowing into `EventEmitter`.
2. **`pgId`** (line 304) is a `number` from `this.pgAgentIds`, which is populated only from a JSON response (`agent.id`) from the dashboard's `CreateAgent` handler. It's an integer from the database — not a string path-injection vector.
3. **`opts.sessionId`** in `pgCreateAgent` at line 274 passes through the same `randomUUID()` origin.

Chi does **not** have `CleanPath` middleware enabled, so path traversal would not be normalized on the receiver side. The finding is technically correct that there's no validation at the caller. But the threat model requires an attacker to control `sessionId`, which requires modifying the in-process `randomUUID()` call or the orchestrator's internal state — at which point you already have code execution.

**Verdict:** Valid as a defense-in-depth hardening measure. `encodeURIComponent()` is trivial and correct. But this is **not a P1** — it's a **P3** code quality improvement. No realistic attack path exists.

#### SEC-2 — API token in memory, no rotation or expiry → **AGREE: P2, correctly scoped**

Confirmed. `this.apiToken` is a plain string, no rotation. The `process.env.MAE_API_TOKEN` exposure via `/proc/self/environ` is real on Linux but requires process-level compromise.

**One additional note the review missed:** The dashboard side (`handlers_pg.go`) loads `tokenMap` once at startup (`LoadTokenMap` at line ~56 of `main.go`) with no refresh mechanism either. So even if the engine rotated, the dashboard wouldn't pick it up without a restart.

**Verdict:** Accurately rated P2. Appropriate for local orchestrator; flagging for production is correct.

#### SEC-3 — No TLS enforcement → **AGREE: P2, correctly scoped**

Confirmed. Default is `http://localhost:8400`. Bearer token sent in cleartext.

**Additional context:** The `corsMiddleware` (line 317) allows configurable origins but defaults wide-open. Combined with no TLS, an on-network attacker could MITM and steal the Bearer token.

**Verdict:** Accurate. P2 is fair — localhost-only mitigates in dev.

#### SEC-4 — Unbounded data in JSON body → **DOWNGRADE: P3 (Low)**

The review correctly identifies that the dashboard's `MaxBytesReader` at `1<<20` (1MB) protects the receiver. The engine would waste bandwidth on oversized payloads.

**However:** The `content` param in `trace()` and `message()` is populated by the orchestrator from agent output. A "compromised or misbehaving agent" that can produce multi-MB content already has arbitrary code execution via the adapter (claude-code, codex, pi all run shell commands). At that point, sending a large payload to the dashboard is the least of your problems.

**Verdict:** Downgrade to P3. Client-side truncation is a good idea for robustness (avoids wasted retry cycles), but not a security concern given the threat model.

#### SEC-5 — Event data fields allow arbitrary injection (XSS) → **AGREE: P3, correctly mitigated**

The review correctly notes that `templ` auto-escapes. I verified: the dashboard uses Go's `templ` library, which HTML-escapes by default.

**Verdict:** P3 is accurate. Defense-in-depth sanitization at the emitter is a nice-to-have but not required.

#### SEC-6 — `dashboardUrl` not validated (SSRF) → **DISAGREE: Should be P3, not P2**

The `dashboardUrl` is set via:
1. Constructor parameter (called from `orchestrator.ts` which reads it from config/env)
2. `process.env` fallback

For this to be exploitable, an attacker needs to control the environment variable or the config file — at which point they can do anything. This is a supply-chain/config-trust issue, not an SSRF in the traditional sense.

**Verdict:** Downgrade to P3. The constructor taking an arbitrary URL is normal. Config integrity is a separate concern.

### Additional Findings Not in Original Review

#### SEC-7 — No buffer size limit (P3 - Low / OWASP A04)

```typescript
private buffer: SessionEvent[] = [];
// ...
this.buffer.push(event);
```

The `buffer` array grows without bound. If the dashboard is unreachable and `fetchWithRetry` keeps returning null, events are drained and dropped (good — `this.droppedEvents++`), so the buffer doesn't actually grow unbounded in the failure case. **On closer inspection this is fine** — `flush()` drains the buffer via `splice(0)` and drops events that fail. Not a real issue.

#### SEC-8 — Race condition in `sessionEnd` ordering (P2 - Medium / Correctness)

```typescript
async sessionEnd(sessionId: string) {
    // ...
    await this.pgUpdateSession(sessionId, { status: "completed" });  // Direct fetch, bypasses buffer
    return this.emit({  // Goes through buffer
      // ...
      event_type: "session_end",
    });
}
```

`pgUpdateSession` makes a direct HTTP call (bypasses the event buffer), while `session_end` goes through `emit()` → `buffer` → `flush()`. If there are buffered events still in-flight, the PG session status gets set to "completed" before earlier events (like cost updates) are flushed. The dashboard may show a completed session with incomplete data.

**This is a real bug, not just a security issue.** The original review mentioned this in the Summary Grades ("pgUpdate-before-emit ordering bug") but didn't elevate it to a finding. It should be.
