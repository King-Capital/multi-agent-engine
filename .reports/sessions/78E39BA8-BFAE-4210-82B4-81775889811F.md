## Summary

Massive multi-agent session that completed Standard Swarm v2 Phase 1 (lifecycle evidence gates) and Phase 2 (participant presence) using three parallel agents (Claude, Pi, Codex) implementing the same scope independently, then cross-reviewing and consolidating. This was the first real test of the multi-agent parallel work protocol that was then codified from the results.

Phase 1 hardened the certification harness with lead-only mode (reducing cert runs from ~$2 to ~$0.35/fixture), chain-runner cert-mode tolerance (degraded steps instead of throwing), canonical artifact selection preferring CERTIFICATION_CONTRACT blocks, and 36 regression tests. All three fixtures (clean, seeded, failing) passed live Pi certification. Phase 2 added participant presence tracking with auto-emit from agentSpawn/agentDone/toolCall/costUpdate, stale detection, and concrete capability metadata (coarse counts instead of raw filesystem globs to prevent info leakage).

The session also established the Cross-Agent Convergence Review Protocol — a formal process for parallel agent work derived from what worked and what didn't. Key finding: parallel agents are valuable not for speed but because independent blind spots overlap less. Each agent found bug classes the others missed (Claude: pipefail bug, Pi: synthesis prompt template, Codex: worker-spawn rejection gate). The protocol was written up in three independent versions, then consolidated. External reviewer provided comprehensive system-level feedback, leading to 4 new GitHub issues (#341-#344) plus 4 more from actionable findings (#346-#349). Auto-version workflow was also fixed to push directly to main instead of creating PRs that required manual merge.

## Decisions Made

1. **Lead-only certification (D-008):** Cert runs spawn only leads, not workers. ~$0.35/fixture vs ~$2. standard-swarm remains the heavy path.
2. **Operational vs substantive failures (D-009):** trace_has_operational_failures checks session crashes only. Agent FAILED grades are substantive outcomes.
3. **Chain-runner doesn't throw in cert mode (D-010):** FEEDBACK grades degrade, FAILED grades still throw. Session completes for synthesis.
4. **Pi as merge base for both phases:** Deepest integration, cleanest architecture. Codex cherry-picked for defensive gates + capabilities. Claude tracker deferred to Phase 6.
5. **Coarse capabilities, not raw globs:** toolCount/readScopeCount/writeScopeCount instead of domain_read/writeGlobs to prevent filesystem layout leakage to dashboard/SSE.
6. **Grade-to-status mapping:** FAILED→failed, FEEDBACK→blocked, others→completed. Previously all non-FAILED became completed.
7. **Auto-version direct push:** Version bumps commit directly to main + tag inline. No more intermediate PRs.
8. **Multi-agent parallel protocol established:** Three agents implementing independently, cross-reviewing, consolidating. Codified as Cross-Agent Convergence Review Protocol.

## Learnings

1. **Three agents independently converging on identical solutions validates correctness.** When three models on different families produce the same fix, the architecture is right.
2. **Each agent has predictable strengths:** Claude finds runtime bugs through execution, Pi builds the cleanest architecture, Codex writes the most defensive tests. Assign roles accordingly.
3. **Worktree discipline matters.** Working in the main repo while others have worktrees creates merge risk. Enforce worktree-only editing.
4. **Handoff detail matters.** Exact trace paths, session IDs, artifact dirs — without them, cross-review requires trust instead of verification.
5. **The pipefail bug pattern** (`grep -q && printf` under `set -euo pipefail`) is a recurring class of shell bug. Always use `if grep; then printf; fi` instead.
6. **Auto-version PRs are an anti-pattern.** Direct push + tag is simpler and avoids the manual merge step.
