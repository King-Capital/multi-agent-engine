## Summary

Massive session that started with dashboard SSE migration, pivoted to Pi adapter rewrite, and ended with a full 5-reviewer swarm audit of the entire codebase. The session exposed fundamental architectural issues: the engine was built as a fire-and-forget batch runner (using `pi --print`) when it should have been using Pi's RPC mode for bidirectional agent communication. The dashboard SSE was broken by Go's `WriteTimeout: 30s` killing all connections after 30 seconds. The `.env` file pointed to the remote dashboard (`MAE_DASHBOARD_URL`) causing all local swarm events to miss localhost.

The Pi adapter was rewritten twice: first from `--print` to `--mode json` (streaming but still fire-and-forget), then to `--mode rpc` (true bidirectional with stdin/stdout JSONL). The orchestrator was gutted from 1096 to ~600 lines, removing dead code (self-healing, security enforcement, prompt building) that Pi's skills and extensions should handle. Worker spawning was added back after initially being removed. Dashboard `main.go` was split from 1211 lines into 4 handler files.

The final swarm review (5 Claude Opus agents) found 10 CRITICAL, 11 HIGH findings. The codebase has real bones (clean adapter abstraction, genuine SSE, working event sourcing) but critical gaps: worktree changes destroyed on cleanup, self-healing/security/budget enforcement all dead code, auth bypass on all GETs, CORS wildcard, parallel session race conditions. The pi-swarm bash harness also ran and confirmed 0/8 passing with 3 FAILED grades.

## Decisions Made

1. **Pi as default adapter over claude-code** — Pi supports 324 models across 20+ providers. Claude-code only runs Anthropic models. The whole point of the engine is cross-model orchestration.
2. **Pi RPC mode over `--print`** — RPC gives bidirectional communication (prompt, steer, follow_up, abort). `--print` is fire-and-forget with no way to inject user messages or steer agents mid-work.
3. **Orchestrator as process manager, not AI agent** — TypeScript handles session lifecycle, worktrees, event routing. Pi agents handle all AI reasoning via their skills. This matches Dan's pattern where the orchestrator IS Pi.
4. **`--local` flag over editing `.env`** — `~/.env` exports `MAE_DASHBOARD_URL` to the remote server. `--local` overrides to localhost without touching any files.
5. **Removed "archived" status** — meaningless lifecycle state. Sessions are done or errored. Filters handle visibility.
6. **`WriteTimeout: 0`** — SSE connections are long-lived. Any positive WriteTimeout kills them. Heartbeats at 15s keep connections alive.
7. **Skills in both `agents/skills/` and `.pi/skills/`** — MAE source of truth at `agents/skills/`, standalone Pi copy at `.pi/skills/`. Two consumers, same content.
8. **Three-tier message hierarchy** — workers→leads→orchestrator→dashboard. Only orchestrator posts to conversation stream. Prevents raw template output from cluttering the UI.

- `self-healing.ts` completely dead — no retry on agent failure

- Markdown not rendering in SSE-appended messages (renderMd may not fire on SSE content)

**Engine:**
- The orchestrator is still a for-loop, not an interactive Pi session (Dan's full pattern)
- `on_feedback` retry logic declared in chains.yaml but never executed
- `till_done` is a progress bar, not a verification gate
- Codex adapter has no streaming, no cost tracking, no timeout

## Learnings

- Go's `WriteTimeout` kills SSE connections silently — set to 0 for SSE servers
- Bun auto-loads `.env` but shell-exported vars take precedence — `--local` flag needed
- Pi's `--mode rpc` is the key to everything — 26+ bidirectional commands, full event streaming
- Pi's `--mode json` gives streaming but is still fire-and-forget (process exits on completion)
- Pi RPC processes stay alive after `agent_end` — must detect and kill explicitly
- `dashboard-bin` binary must be rebuilt after Go changes (easy to forget when iterating)
- The IndyDevDan research at `.reports/indydevdan-research/` has full video analyses with architectural patterns
