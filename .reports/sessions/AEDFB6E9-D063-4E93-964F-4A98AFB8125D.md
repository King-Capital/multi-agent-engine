## Summary

This session moved MAE from late RC cleanup into a deployed `v1.0.4` state. The main thread was release hardening around real operational problems: installer parity across hosts, optional A2A health, stale session lifecycle, wrapper self-update corruption, dashboard cost/status mismatches, and making sure Skippy and King boxes actually run the same MAE config as local.

The most important fixes were the wrapper update path and stale session handling. `mae update` had been copying over the wrapper file while that same shell script was still executing, which caused corrupted mid-run reads like `xy: command not found`. The fix updates through a same-directory temp file and atomic move, preserving symlink installs. The stale session reaper now closes open stale sessions as `completed`/done instead of incorrectly marking them as `error`; it also covers `active`, `waiting`, and `paused` sessions in PostgreSQL.

Current state is good: local main is clean and synced to `v1.0.4`, live dashboard health reports `version: 1.0.4` with DB connected, and both Skippy (`10.71.1.21`) and King (`king@10.71.20.120`) are installed at `v1.0.4` with MAE health `HEALTHY`, Langfuse connected, and all three adapters available (`echo`, `pi`, `a2a`).

## Decisions Made

- Treat A2A as optional unless `MAE_A2A_URL` is configured because no MAE deployment should be degraded only because remote A2A delegation is not in use.
- Use MonkeyProof A2A (`10.71.20.120:3210`) as the MAE coding-delegation A2A target for now; Bilby A2A remains infra-specific.
- Keep the installer/wrapper update atomic rather than trying to run `mae update` from the old installed wrapper on remote boxes.
- Close stale sessions as `completed` rather than `error` because stale here means the dashboard/session lifecycle missed terminalization, not necessarily that the run failed.
- Keep PR-sized fixes narrow and deploy each after merge; this made the release bumps `v1.0.2`, `v1.0.3`, and `v1.0.4` easy to verify independently.

## Known Issues / Blockers

- Local Codex sandbox health checks can falsely report trace/dashboard/Langfuse problems because this process cannot write `/Users/rico/.mae/traces` and may not have the same network/env as the actual user shell. Remote and live checks are more authoritative.
- Live `close-stale` was not manually POSTed after deploy because it would close all current open sessions. Behavior is covered by Go tests and deployed code inspection.
- The Skippy scout timeout/nudge issue was mitigated by PR #241 but still deserves a real controlled run on a large repo with Pi heartbeat/orchestrator-loop behavior watched.
- The briefing from 2026-05-10 still listed #192 as a P0 blocker. That is now functionally addressed by PR #243, but issue/board state may need manual cleanup if GitHub issue metadata was not auto-linked.

## Learnings

- Updating a shell wrapper by copying over the file currently being interpreted can corrupt execution mid-run. Use temp file plus atomic rename for self-updating scripts.
- Symlink-preserving installs need full symlink-chain resolution, not one-hop handling.
- Dashboard stale cleanup had two separate stores to keep aligned: in-memory JSONL event store and PostgreSQL. Fixing only one leaves confusing UI/state mismatches.
- `GOCACHE=/tmp/mae-go-cache` is needed for Go tests from this Codex sandbox when the default macOS Go build cache is not writable.
- Health should be judged from the actual MAE environment on each host, sourcing `~/.mae/config` and `~/.cc-env`, not from a stripped non-interactive sandbox shell.
