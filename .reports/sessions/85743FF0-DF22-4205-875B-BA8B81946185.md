## Summary

Major feature + infrastructure session that shipped issue #180 (Frontend Design Specialist), built out the MAE standard swarm architecture with 5 specialist squads of 6 workers each, extracted duplicated stream handler code, ran multiple swarm reviews (both CC standard and MAE multi-agent), and fixed ~20 security/correctness/engine issues discovered during review.

The design specialist delivers a complete workflow: a designer persona with deep CSS/typography/spacing domain knowledge, a `Bun.serve()` design gallery that live-previews generated HTML variants with path traversal/XSS/CSP/SSRF protections, a reference loader that pulls design context from files/URLs/project scans, and a `mae design` CLI command with three modes (interactive session, design-review chain, design-build chain). A new Design team was added to teams.yaml with a planner lead, designer worker, frontend engineer, and antagonist.

The standard swarm evolved significantly during the session. The initial implementation reused existing Red/Blue teams (2 teams, ~5 agents). After observing this on the live dashboard, the user correctly identified that a proper swarm needs individual specialist squads — not pre-composed teams. The final architecture has 5 squads (Correctness, Adversarial, Quality, Security, Domain) each with a validator/planner lead and 6 specialist workers across multiple model families (Opus, Sonnet, GPT-5.5). Leads are prompted with `system_prompt_append` in SWARM MODE to brief all workers immediately, stream progress updates, and relay findings in real-time.

Several critical engine fixes emerged from the live swarm testing: the orchestratorLoop was completely blind to team-execution activity (never wired through TeamExecutionDeps), leads were marked "done" immediately after briefing their workers (before workers even spawned), the stall detection threshold was too aggressive (90s → 180s) for large swarms, integration tests hardcoded localhost:8400 instead of using MAE_DASHBOARD_URL, and sessions never auto-closed when work completed. Six new GitHub issues (#187-192) were filed for dashboard UX and engine lifecycle improvements needed for production.

## Decisions Made

1. **Bun.serve() for design gallery, not Go dashboard** — the gallery serves arbitrary agent-generated HTML that needs live preview during design sessions. Integrating into the Go dashboard would require iframe sandboxing and new API routes. Standalone Bun.serve() is 137 lines, hot-reloads on file change, and can be killed when the session ends. Dashboard gallery view filed as future follow-up.
2. **5 specialist squads over Red/Blue team reuse** — user observed on live dashboard that the swarm-review chain (Red+Blue) only produces 2 perspectives with pre-composed teams. A proper swarm needs independent specialist leads each spinning out their own workforce. 5 squads x 6 workers = 30 agents with cross-model coverage.
3. **Leads stay running until session close** — leads were marked "done" after writing briefs (before workers spawned). This broke the dashboard's ability to show lead status during worker execution. Moving `agentDone` to the end of the team step (after worker completion + review + retries) keeps leads visible as active supervisors.
4. **orchestratorLoop must be piped through TeamExecutionDeps** — was completely absent from team-execution.ts, making the orchestrator loop blind to all team activity. Only solo `runAgent` calls registered events. Now all agents (leads, workers, synthesis) report to the loop.
5. **system_prompt_append SWARM MODE for leads** — rather than creating new lead personas, the standard-swarm chain overrides the lead behavior via `system_prompt_append` in the chain config. Leads brief all workers immediately, stream progress, and relay findings as they come in. This keeps the persona files generic and the swarm behavior in the chain definition.
6. **180s stall threshold over 90s** — with 30+ agents spawning in parallel, leads take 60-90s just to read files before making their first tool call. 90s threshold was generating constant false stall detections. 180s gives breathing room without masking real stalls.
7. **Image references as metadata only, not base64** — swarm reviewers caught that reading full image binaries into memory then truncating base64 to 200 chars was both an OOM risk and useless to the LLM. Changed to pass file path + size + type metadata only.

## Known Issues / Blockers

1. **Sessions never auto-close** (#192) — when a chain completes and the CLI exits, the dashboard session stays "active" forever. Needs `session.status = "completed"` in chain-runner's finally block. Ghost sessions accumulate.
2. **Orphaned Pi processes on session close** — dashboard "close session" just flips a DB field. The Pi adapter's spawned CLI processes (Claude Code, Codex) continue running as zombies. Need process tracking + SIGTERM on session close.
3. **Worktrees don't include uncommitted files** — the standard-swarm creates git worktrees for each squad, but uncommitted new files aren't in the worktree. The Correctness Squad lead couldn't find any files to review. Fix: commit before running swarm, or copy uncommitted files to worktrees.
4. **Dashboard agent tree has no team grouping** (#189) — 30+ agents render in a flat row. Can't tell which workers belong to which lead without clicking each one.

## Learnings

1. `Bun.file().toString()` returns `[object Blob]`, not file contents — use `readFileSync()` or `await Bun.file().text()`. Caught by 4 of 5 swarm reviewers.
2. Running multiple CC agents as background reviewers is fast (~90s each) but the findings need dedup — 3 reviewers found the same path traversal bug independently.
3. The MAE swarm on its own code is a powerful meta-test. Observing 28 agents on the dashboard immediately revealed UX gaps (agent grouping, clickability, lead status) that wouldn't surface from unit tests.
4. `system_prompt_append` on chain steps is the right lever for swarm behavior — keeps personas generic and specialization in the chain config.
