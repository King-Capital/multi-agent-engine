Evidence: `agents/teams/chains.yaml:270-281`, `engine/chain-validator.test.ts:72-96`.
- Correct: Runtime review-only propagation exists for parallel teams: `buildParallelTeamStep` carries `read_only` from team or parent step (`engine/team-execution.ts:832-845`), `runParallelStep` disables worktree creation if any parallel child is review-only (`engine/team-execution.ts:874`), and lead/worker delegate options are restricted to read/grep/glob with write/update domains cleared (`engine/team-execution.ts:250-265`, `engine/team-execution.ts:424-436`; helper in `engine/review-mode.ts:3-17`).
- Correct: The certification script now isolates traces with `MAE_TRACE_DIR="$trace_dir"` instead of inspecting global `~/.mae/traces`, and verifies the final session trace exists before passing (`scripts/certify-live-swarm:37-40`, `scripts/certify-live-swarm:107-110`, `scripts/certify-live-swarm:163-173`).

- Blocker: `scripts/certify-live-swarm` is very likely inspecting the wrong artifact for the actual swarm-review result. `review_report_path` only looks for `pi-orchestrator-output-*.txt` (`scripts/certify-live-swarm:86-98`), but current artifact names are generated from the runtime `agentId` (`engine/trace-artifacts.ts:31-37`). For a parallel swarm, synthesis uses `synth-${session.id.slice(0, 8)}` (`engine/team-execution.ts:958-965`) and the emitted artifact is therefore `synth-...-output-*.txt`, not `pi-orchestrator-output-*.txt` (`engine/team-execution.ts:1029-1034`). If no matching artifact is found, the script falls back to CLI stdout (`scripts/certify-live-swarm:94-98`), which may only contain summarized output rather than the full review report. Immediate fix: resolve the artifact path from the trace `agent.end` event for the synthesis/final agent, or search all `*-output-*.txt` artifacts and prefer synthesis/final outputs.

- Blocker: The live certification can pass with partial swarm execution because failed parallel teams are only downgraded to `FEEDBACK`, not a hard failure. `runParallelStep` converts rejected teams into synthetic failed results and continues synthesis (`engine/team-execution.ts:926-952`); after synthesis, it changes the grade to `FEEDBACK` when any team failed (`engine/team-execution.ts:1017-1024`). The certification script does not check command/session grade or failed-team markers, only content regexes (`scripts/certify-live-swarm:146-160`). That means a 4-of-5 review with one dead squad can still satisfy the broadened YAML regexes and print `CERTIFICATION PASS`. Immediate fix: certification should fail on any `worker_failed`/failed parallel team event or non-`PASS`/non-`VERIFIED` final grade for live-pi.

- Note: The new `swarm-review` till_done gates are intentionally broad enough to be brittle as production readiness evidence. For example, the “scope and verdict” gate accepts generic `PASS`, `CLEAN`, or `VERIFIED` anywhere after `Scope|File|Fixture|Teams` (`agents/teams/chains.yaml:283-285`), and “five canonical perspectives” can pass on the phrase `five canonical perspectives` without listing all five perspectives (`agents/teams/chains.yaml:286-288`). Because `verifyTillDone` applies these regexes to only the first 10,000 chars of the combined output (`engine/chain-runner.ts:141-149`), long multi-team outputs can also fail/pass based on truncation position. Immediate fix: add deterministic tests for representative clean/seeded/failing outputs and tighten gates around explicit schema blocks from all five teams plus synthesis.

- Note: There is no dedicated unit coverage for `review-mode.ts` behavior. The existing validator tests assert YAML shape, but do not prove `readOnlyTools`, domain clearing, or `buildWorkerSystemPromptAppend` preserve the right review instructions (`engine/review-mode.ts:3-42`). Add focused tests, especially for tool fallback and ensuring mutation tools stay excluded.
