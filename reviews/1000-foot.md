- Blocker: Certification is not yet a release gate. `scripts/certify-live-swarm` is not referenced by the deploy/release workflows inspected; deploy only checks dashboard health after engine sync (`.github/workflows/deploy.yml:155-158`), and docs release gate only asks for dry-run `mae chain review-only` plus score (`docs/release-readiness.md:24-35`). Recommended next goal: add a required certification job that runs `scripts/certify-live-swarm` in echo mode for PR/CI and `scripts/certify-live-swarm --live-pi --dashboard-url <prod/internal URL>` before production release/deploy promotion.
- Blocker: The changed certification script defaults to localhost (`scripts/certify-live-swarm:18-19`). That is good for local smoke but weak as “production readiness” unless CI/prod explicitly passes the deployed dashboard URL. Recommended next goal: make the deploy/release gate pass the actual dashboard URL and record artifacts/session IDs on failure.
- Blocker: Echo-mode certification does not validate review intelligence. In non-Pi mode the seeded fixture only checks that the vulnerable fixture file contains `exec("nslookup " + name)` (`scripts/certify-live-swarm:154-155`), not that MAE found it. Recommended next goal: keep echo as plumbing smoke, but label it non-security-certifying in CI output and require Pi mode for any “production certified” verdict.
- Blocker: Dashboard health/deploy checks are shallow. Deploy only curls `/api/health` (`.github/workflows/deploy.yml:155-158`), while production expectations include auth, RBAC, SSE replay, event persistence, SPA, and token handling. Recommended next goal: add a dashboard smoke script covering login/session cookie or bearer token, `/api/events`, `/api/stream`/session stream, admin-denied mutation for non-admin, and SPA 200.

- Note: Several chain gates remain regex/string-output based, so certification can pass on phrasing rather than structured data. Example: `swarm-review` accepts broad alternatives like `PASS`, `CLEAN`, `VERIFIED`, and “five review dimensions” (`agents/teams/chains.yaml:283-294`). Recommended next goal: converge the final synthesis to a strict machine-readable appendix and gate on all five raw `REVIEW_REPORT` blocks, explicit verdict, and P0/P1 counts.
- Note: Security scanning is present but narrow. CodeQL is configured (`.github/workflows/security-scan.yml:43-60`), and `.gitignore` references ggshield cache, but I did not find workflow evidence for dependency vulnerability scanning or secret scanning as required release gates. Recommended next goal: add Bun/Go dependency audit and secret scan to PR gates, with pinned actions and SARIF upload where supported.
- Note: Release docs lag the new certification approach. `docs/release-readiness.md:24-35` does not mention live Pi certification, dashboard RBAC/SSE smoke, or security scan status. Recommended next goal: update the release checklist so “ready” requires tests/typecheck/build, CodeQL/dependency/secret scans, dashboard smoke, and live Pi swarm certification evidence.
- Note: Current branch diff is focused: chain read-only flags and softer `swarm-review` regexes in `agents/teams/chains.yaml`, plus certification harness hardening in `scripts/certify-live-swarm`. The main risk in the diff is that softer regexes reduce false negatives but can mask incomplete reports unless paired with stricter artifact parsing.
