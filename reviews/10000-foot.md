# 10000-foot MAE Production Readiness Audit

## Review

- Correct: **Mission is clear and differentiated.** MAE is positioned as a multi-agent orchestration harness for shipping code with model-diverse planning, implementation, validation, and review. README states the product promise as “Ship code with an army of AI agents” and lists multi-model orchestration, cross-model verification, real-time dashboarding, sandbox pooling, pipeline resume, cost tracking, and guardrails as core features (`README.md:24-43`). The architecture diagram frames a sensible control plane: user/CLI → orchestrator → planning/engineering/validation teams, with dashboard event streaming (`README.md:53-79`).

- Correct: **The project has moved beyond prototype shape into a real v1 product surface.** The docs declare stable release `v1.0.20` and explicitly scope shipped adapters to `pi`, `a2a`, and `echo`, while excluding Claude Code/Codex until smoke coverage exists (`README.md:24-30`, `docs/release-readiness.md:1-18`). The repository contains a substantial engine, Go dashboard, React dashboard, chain/team YAML, CI workflows, release-readiness gates, observability docs, and 552 passing engine tests from `cd engine && bun test`.

- Correct: **Architecture maturity is strongest in the engine core and workflow model.** Evidence: chain definitions encode plan-build-review, review-only, full-SDLC, swarm review, red/blue, scout-swarm, and build-then-swarm flows (`agents/teams/chains.yaml`); model routing encodes role tiers, cross-model pairs, budget ceilings, and concurrency limits (`configs/model-routing.yaml:1-80`); review-only enforcement exists as first-class code by restricting tools and writable domains (`engine/review-mode.ts:1-27`). This is a meaningful control-plane design, not just prompt scripts.

- Correct: **Release hygiene exists and is testable.** The release readiness document requires tests, typecheck, dashboard build, Go build, CLI build, version check, dry-run smoke, trace events, and score validation (`docs/release-readiness.md:20-35`). Local verification in this audit: `cd engine && bun test` passed with `552 pass, 1 skip, 0 fail`; `cd engine && bunx tsc --noEmit` exited successfully.

- Note: **Current branch is focused on hardening live swarm certification, which implies the highest-risk production surface is not generic code quality but end-to-end trustworthiness of live agent review.** Branch: `fix/live-certification-hardening`. Uncommitted changes include read-only flags in review chains and major changes to `scripts/certify-live-swarm`; `git diff --stat` shows 11 changed files, including `agents/teams/chains.yaml`, `scripts/certify-live-swarm`, and several `.pi/skills/*` files. The diff tightens review-only configuration and relaxes/clarifies swarm output matching around unified certification reports (`agents/teams/chains.yaml:260-292`). Strategic read: MAE is converging on “can we trust swarms to certify work?” rather than “can we run agents at all?”

- Blocker: **Production readiness is not yet proven end-to-end for the live system.** The strongest evidence available is unit/integration-style engine coverage and dry-run/release gates, not a completed live Pi certification run. The release gate itself requires dashboard build, Go build, `just build`, `mae version`, dry-run release smoke, trace scoring (`docs/release-readiness.md:20-35`); this audit only verified engine tests and TypeScript. The current branch introduces a live certification harness, but its presence and active hardening indicate certification is still being made reliable, not already institutionalized.

- Blocker: **The roadmap still lists major production-hardening capabilities as open.** Phase 6 explicitly leaves unchecked: self-hosted runner auto-deploy, `/healthz` with PG connectivity, graceful shutdown/drain, session recovery from PG, agent timeout enforcement, hard budget stops, Prometheus/Grafana monitoring, structured log aggregation, and alerting (`docs/ROADMAP.md:83-94`). Those are not polish items; they are operational prerequisites for confidently running MAE as shared production infrastructure.

- Note: **Dashboard/runtime persistence posture is partially mature but not yet a complete reliability story.** The dashboard initializes file-backed session storage by default (`dashboard/main.go:76-83`), reaps stale sessions on startup (`dashboard/main.go:87-90`), and attempts PostgreSQL initialization with fallback behavior visible in code immediately after the inspected region. This supports degraded local operation, but the roadmap still calls out PG health, graceful shutdown, session recovery, metrics, and alerting as future work (`docs/ROADMAP.md:85-94`).

- Note: **The biggest systemic risk is confidence inflation: docs and release status say v1 stable, but certification and operations are still catching up.** README advertises sandbox pooling, pipeline resume, cost tracking, and damage-control rules (`README.md:40-43`), while roadmap Phase 6 still has unchecked operational controls for timeouts, budget hard stops, recovery, monitoring, and alerting (`docs/ROADMAP.md:83-94`). This does not mean the claims are false; it means production readiness depends on narrowing the gap between feature existence, test coverage, live certification, and operational runbooks.

- Note: **Second systemic risk is agent authority control.** The current branch’s read-only edits are strategically important because review chains previously needed explicit hardening to ensure reviewers do not mutate code. Code-level support exists (`engine/review-mode.ts:1-27`), and the branch extends read-only flags across review-style chains. The remaining strategic question is whether this policy is comprehensively verified across Pi/A2A/live adapters, not just represented in YAML and unit tests.

- Note: **Third systemic risk is observability-to-ratchet incompleteness.** The repo has trace recording, Langfuse/Ralph docs, and goals for judge/ratchet work, but the high-level readiness picture depends on closing the loop: live traces → scoring → judged regressions → accepted/rejected config mutations. The goals docs frame this as still being built: Ralph is described as advisory without a safety-net ratchet in `docs/goals/goal-2-ralph-ratchet-and-judge.md`, and production-grade Langfuse/Ralph is a stated goal in `docs/goals/langfuse-observability-and-ralph-ratchet.md`.

## Highest-level truth

MAE is best described as a **credible v1 multi-agent orchestration platform in late production-hardening**, not as a fully operationally proven production service. Its mission and architecture are coherent; the engine has substantial tests; release gates exist; and the current branch is addressing the right strategic risk: live swarm certification trust. The remaining gap is not “write more features.” It is to prove, repeatedly and automatically, that live swarms can review safely, produce trustworthy severity output, preserve artifacts/traces, enforce authority boundaries, respect budgets/timeouts, and recover/alert under operational failure.

## Recommended next goal set

1. **Production Certification Goal:** make `scripts/certify-live-swarm --live-pi` the canonical release blocker, with clean/seeded/failing fixtures, artifact preservation on failure, trace scoring, and dashboard evidence.
2. **Operational Control Goal:** close Phase 6 essentials first: health including PG, graceful shutdown, session recovery, timeout enforcement, hard budget stops, metrics, and alerting.
3. **Authority Boundary Goal:** prove review-only/write-domain/tool restrictions across every shipped adapter (`pi`, `a2a`, `echo`) with live or adapter-level tests.
4. **Evidence Pipeline Goal:** require every production run to emit durable trace artifacts, cost data, chain step events, final verdict, and scoreable outcomes.
5. **Ratchet Goal:** convert Ralph from advisory improvement into gated improvement: replay golden traces, judge quality, reject regressions, and record accepted mutations.
6. **Docs Truth Goal:** align README/release/status claims with what is live-certified, what is dry-run-only, and what is roadmap.
